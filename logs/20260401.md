## 四、Stage 4-B：3_coding.py xKG 注入改造（已完成）

### 改动说明

文件：`Paper2Code/codes/3_coding.py`

**改动 1**：新增 CLI 参数 `--xkg_path`（第 20-22 行）

```python
parser.add_argument('--xkg_path', type=str, default=None,
                    help='Path to validated xKG JSON (output of validate_xkg.py). '
                         'If provided, injects verified code nodes as pre-written files.')
```

**改动 2**：变量赋值（第 34 行）

```python
xkg_path = args.xkg_path
```

**改动 3**：注入逻辑（第 58-72 行，插入在 `done_file_dict = {}` 之后）

```python
# xKG 注入：将 validate_xkg.py 验证过的核心代码节点作为已完成文件注入 LLM 上下文
if xkg_path:
    with open(xkg_path, encoding='utf-8') as f:
        xkg_data = json.load(f)
    os.makedirs(output_repo_dir, exist_ok=True)
    for tech in xkg_data.get('techniques', []):
        if not tech.get('verified'):
            continue
        filename = tech['name'].lower().replace(' ', '_') + '.py'
        code = tech['code']['implementation']
        done_file_dict[filename] = code
        done_file_lst.append(filename)
        with open(os.path.join(output_repo_dir, filename), 'w', encoding='utf-8') as f:
            f.write(code)
    print(f"[xKG] Injected {len(done_file_dict)} verified node(s): {list(done_file_dict.keys())}")
```

### 机制说明

- `--xkg_path` 缺省（`None`）时，脚本行为与原版完全一致（向后兼容）
- 注入后，`done_file_dict` 已含 xKG 验证代码，`done_file_lst` 已含对应文件名
- 后续每次 LLM 生成新文件时，`get_write_msg()` 会把 `done_file_dict` 里的内容拼入 prompt 的 `## Code Files` 段——LLM 生成 `trainer.py`、`dataloader.py` 等时自然可见核心模型代码，避免凭空幻觉
- 注入的文件同时被物理写入 `output_repo_dir/`，确保最终仓库中该文件存在

### 调用方式

```bash

    --gpt_version   gpt-4.1-mini \
    --paper_format  JSON \

    --output_dir    outputs/repos/AttentionCalibration \
```

## 六、Stage 5：pipeline/refine_repo.py（已完成）

### 设计说明

直接调用 aider Python API（本地 `aider/` 源码，加入 `sys.path`），无需 CLI subprocess。

```python
coder = Coder.create(
    main_model=Model(model_name),
    fnames=[...所有 .py 文件...],
    io=InputOutput(yes=True),   # 非交互模式
    auto_commits=False,
)
coder.run(fix_prompt)
```

aider 会自动生成 Repo Map（语言解析器分析所有文件的 symbol tree），让 LLM 在单次上下文中看到全仓库符号关系，实现跨文件一致性修复。

### 修复目标（_FIX_PROMPT）

1. Import 错误（未定义名称、错误模块路径）
2. 接口不一致（方法签名/参数名不匹配）
3. 未定义的 class/function 引用（添加最小 stub）
4. 循环导入
5. config.yaml 中不存在的 key 读取

### 调用方式

```bash
python pipeline/refine_repo.py \
    --repo_dir outputs/repos/AttentionCalibration/repo \
    --model    gpt-4.1-mini \
    --rounds   2
```
